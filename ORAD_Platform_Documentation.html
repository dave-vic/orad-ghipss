Section 01

Overview

What ORAD is and why it was built

ORAD (Operations Restricted Access Directory) is a secure, internal document management platform built for GhIPSS. It provides a single, controlled location for storing, accessing, and auditing sensitive operational documents — with access governed by role and folder-level permissions.

The platform replaces ad-hoc file sharing (shared drives, email attachments) with a structured system that tracks who accessed what and when, enforces access restrictions, and gives administrators visibility and control over the entire document lifecycle.

Secure Document Storage

Documents are stored server-side with controlled access. No public URLs — every download is authenticated.

Role-Based Access Control

Three roles (Admin, Member, Viewer) with configurable permissions per folder and document type.

Full Audit Trail

Every login, download, upload, deletion, and role change is logged with user, timestamp, and IP address.

Access Request Flow

Users can request access to restricted folders. Admins review, approve or deny with optional notes.

Section 04

Pages & Features

A walkthrough of every screen in the platform

Login

The entry point to the platform. Users authenticate with a username and password. A JWT token is issued and stored in an HTTP-only cookie. Demo credential cards are displayed for quick testing access. On success, the user is redirected to the Dashboard.

Dashboard

The home screen after login. Content adapts based on the user's role.

Admin view includes:

  • Insight alert banners (pending access requests, stale documents) — dismissible, and also reflected in the notification bell
  • Favourited folders and documents as quick-access pills
  • Stat cards: Total Documents, Active Users, Downloads (30 days), Uploads (30 days), Pending Requests
  • Most Downloaded documents in the last 30 days
  • Recent Activity feed showing all user actions
  • Documents not accessed in 90+ days with a direct link

Member / Viewer view includes:

  • Downloads stat card (personal)
  • Personal recent activity feed
  • Favourites

Directory

Browse all folders in the platform. Folders the user can access are shown as full cards with a custom icon. Restricted folders (no access) are shown in a locked state — clicking them opens an access request form. Accessible folders navigate into the Folder Page on click.

Folder Page

The core document management screen. Displays all documents within a folder in a structured table.

ColumnDetails
Document nameFile icon, name, tag badges
TypeFile extension (PDF, DOCX, etc.)
SizeHuman-readable file size
UploadedUpload date
Uploaded byUser avatar and name
ActionsView, Download, Comment icons. Admin gets additional kebab menu.

Additional features on this page:

  • Document viewer (PDF and image preview in a full modal)
  • Comments panel per document
  • Version history — upload a new version, view previous versions
  • Guest link sharing — create a time-limited, view-count-limited public link
  • Folder announcement banner — admin can post a notice shown to all users
  • Require Acknowledgement — mark a document as requiring user sign-off
  • Admin: Edit folder, Delete folder from the ··· menu

My Activity

Personal history for the logged-in user, split into two tables:

  • Recently Accessed — documents the user has viewed or downloaded, with action badge and date
  • My Acknowledgements — documents the user has formally acknowledged, with folder and date. Rows navigate to the relevant folder.

My Requests

For Members and Viewers to track their folder access requests. Shows request status (Pending / Approved / Denied) and any reviewer note left by the admin.

Activity Log

Admin-only. A complete, filterable audit trail of every action taken on the platform — login, logout, download, upload, delete, role change, folder access grant/revoke, comments, guest links, and more. Exportable to CSV.

User Management

Admin-only. Two-tab interface:

Users tab

  • Table of all users: avatar, name, email, role badge, active status toggle, joined date, last login
  • Search, sort, filter
  • Bulk select + bulk actions (deactivate, delete)
  • Per-user kebab menu: View Profile, Reset Password, Change Role, Deactivate/Activate, Delete
  • View Profile modal with full user details
  • Add User modal to create new accounts
  • Bulk Import via CSV file upload
  • Pagination (10 / 20 / 50 rows)

Roles & Permissions tab

  • Visual matrix of all permissions across all roles
  • Collapsible categories: Documents, Folders, Activity & Logs, User Management, Settings
  • Inline toggle checkboxes (Member and Viewer editable; Admin locked)

Access Requests

Admin-only. Review folder access requests submitted by Members and Viewers.

  • Tabbed by status: Pending / Approved / Denied
  • Table showing: User, Folder, Reason, Submitted date, Status
  • Review modal: shows requester details and reason, Approve / Deny toggle, optional note to the user
  • Approving a request immediately grants the user access to that folder

Account Settings

Accessible from the top-right user dropdown. Allows the logged-in user to update their display name, email, change their password, set up two-factor authentication, and view/revoke their active sessions.

Notifications

The bell icon in the top bar shows unread notification count. Clicking it opens a dropdown panel with all notifications. Admin notifications include daily insight alerts (pending requests, stale documents). Notifications are clickable and navigate to the relevant page.

Section 07

Security

How the platform protects data and controls access

Authentication

Users authenticate with a username and password. On success, the server issues a signed JWT stored in an HTTP-only cookie — this means the token is never accessible to JavaScript and cannot be stolen via XSS. Every protected API route requires a valid token verified by the authenticate middleware.

Passwords

All passwords are hashed using bcrypt with 10 salt rounds before being stored. Plain-text passwords are never saved to the database.

Role Enforcement

Admin-only routes are protected by a requireRole('admin') middleware that checks the user's role from their JWT payload and returns a 403 if the check fails.

CORS

The API only accepts requests from the configured frontend origin (CLIENT_URL in the server .env). All other origins receive a CORS error.

Session Management

Sessions are tracked in the database (UserSession model) with a unique JWT ID (jti). Admins can view and revoke any active session. Revoked sessions are rejected even if the JWT has not yet expired.

Guest Links

Shared document links use a random UUID token. They can be set to expire by date and/or by view count. No authentication is required to access a guest link, but they are rate-limited and tracked.

Audit Logging

Every significant action is logged: logins (success and failure), logouts, downloads, uploads, deletions, role changes, folder access grants/revocations, comments, guest link creation, 2FA changes, and session revocations.

Note: File storage currently uses the local filesystem. For production, files should be migrated to an object storage service (e.g. AWS S3 or Google Cloud Storage) to ensure durability and scalability.
Section 08

Running the Platform

How to start both the frontend and backend locally
Both the backend server and frontend dev server must be running at the same time in separate terminals.

Prerequisites

  • Node.js 18 or higher
  • PostgreSQL running locally
  • .env file configured in /server with DATABASE_URL and CLIENT_URL

Start the Backend

cd server
node src/index.js

Server runs on http://localhost:4000

Start the Frontend

cd client
npm run dev

Frontend runs on http://localhost:5174

First-Time Setup

cd server
npx prisma migrate dev   # Creates all database tables
node prisma/seed.js      # Creates demo users and folders
